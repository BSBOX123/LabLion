#  홈랩 프로젝트 개발 일지 (v1.0)

##  1. 핵심 Q&A 및 기술 인사이트 

*   **Q. MySQL 대신 MariaDB를 선택한 이유는?**
    *   **A.** MariaDB는 MySQL의 '이란성 쌍둥이'로 문법과 포트가 99% 동일함. 하지만 오라클의 무거운 기능을 덜어내어 **라즈베리파이(ARM/리눅스 환경)에서 훨씬 가볍고 빠르게 동작**하기 때문에 소규모 홈랩에 최적화된 선택임.
*   **Q. 도커 실행 시 `address already in use` (포트 충돌) 에러의 원인과 해결책은?**
    *   **A.** 내 컴퓨터의 기존 프로그램이 3306번 포트를 점유하고 있을 때 발생. 기존 프로그램을 지울 필요 없이, 도커의 포트 매핑 기능(`-p 3307:3306`)을 활용해 우회 접속로를 뚫어주는 방식으로 10초 만에 해결함.
*   **Q. 대기업 실무에서도 도커로 DB를 구축하는가?**
    *   **A.** **개발/테스트 환경에서는 100% 도커를 사용함** (환경의 일관성 유지). 단, 실제 운영(Production) 환경에서는 데이터 유실을 막기 위해 AWS RDS를 쓰거나, 최소한 **'볼륨 마운트(Volume Mount)'** 기술을 적용하여 컨테이너가 파괴되어도 하드디스크에 데이터가 남도록 설계함.
*   **Q. 도커(Docker)가 곧 클라우드(Cloud)인가?**
    *   **A.** 도커 자체는 클라우드가 아니지만, 현대 클라우드를 만드는 '마법의 상자(캡슐화)' 기술임. 내 노트북이나 라즈베리파이에 도커를 깔고 인프라를 올리는 순간, 그 기계는 단순한 PC가 아니라 **'나만의 프라이빗 클라우드 서버'**가 됨.
*   **Q. 탐색/정렬 알고리즘을 직접 짜서 데이터를 관리해야 하는가?**
    *   **A.** **절대 아님.** DB(MariaDB) 내부에 이미 인류 최고 수준의 B-Tree 탐색 및 정렬 알고리즘이 내장되어 있음. 백엔드 개발자는 직접 알고리즘을 짜는 수학자가 아니라, SQL과 비즈니스 로직을 이용해 이 거대한 톱니바퀴들을 조립하고, 트래픽 폭주 시 데이터가 꼬이지 않게 제어(트랜잭션)하는 '지휘자'임.
